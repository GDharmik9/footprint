Here is the complete infrastructure mapping for the **Footprint** application, broken down by Google Cloud (GCP) components and the specific third-party APIs required to power the tracking engines.

---

## 1. Google Cloud Services Architecture

To minimize operational overhead and handle asynchronous data pipelines efficiently, the architecture leans heavily on GCP’s serverless and managed services.

### Compute & API Layer

* **Google Cloud API Gateway:** Act as the secure entry point for the mobile app client. It handles authentication routing, rate limiting to prevent API abuse, and log aggregation.
* **Cloud Run (Serverless Containers):** Powers our microservices (Ingestion, User Service, Rewards Engine). Cloud Run automatically scales to zero when there is no traffic (saving costs) and scales up instantly during high-traffic push notification windows.

### Asynchronous Data Pipeline & Workers

* **Cloud Pub/Sub (Event Bus):** The central nervous system of the app. When raw data streams arrive from a utility webhook or transit tracking SDK, they are published as events (e.g., `utility.ingested` or `trip.completed`) to be processed asynchronously by the calculation engine.
* **Cloud Tasks:** Manages rate-limited queues for calling external APIs. When polling thousands of utility accounts via Arcadia, Cloud Tasks ensures we don’t hit rate limits or timeout errors.
* **Cloud Scheduler:** A managed cron service that triggers daily or hourly batch jobs (e.g., prompting the Ingestion Service to pull new data or evaluating weekly Eco-League brackets).

### Storage & Caching Layer

* **Cloud SQL for PostgreSQL:** Stores relational, structured data such as user account details, authentication references, gamification levels, streaks, and corporate sponsor contract data.
* **TimescaleDB on Compute Engine (or Bigtable):** For long-term storage of time-series carbon data points. If using an entirely managed GCP option, **Cloud Bigtable** handles millions of incoming raw IoT or transit telemetry metrics effortlessly.
* **Memorystore for Redis:** A fast, in-memory cache used to store regional grid emission factors (e.g., mapping zip codes to carbon intensity). This prevents our calculation microservice from making slow, costly external API requests every single time a user logs an action.

### Security & Compliance

* **Secret Manager:** Securely stores sensitive credentials, database connection strings, and the master OAuth tokens used to access third-party APIs like Nest or Arcadia. All tokens are encrypted at rest using envelope encryption.

## 2. Third-Party APIs & Integration Points

To populate the tracking dashboards with zero or low-friction manual data entry, the backend connects to three core API pillars:

### A. Energy & Smart Home Tracking

* **Arcadia API (or Urjanet):** The primary engine for utility data collection. It functions like Plaid but for energy bills. Users authenticate their local power utility once, and Arcadia provides clean JSON data detailing monthly kilowatt-hours ($kWh$) and gas usage ($therms$).
* **Google Nest Device Access API:** Connects directly to smart home ecosystems. It allows the backend to securely read HVAC runtime data, ambient indoor temperature configurations, and eco-mode statuses to verify smart home challenges.

### B. Transit & Travel Tracking

* **Radar.io SDK & API:** Handles mobile context and geofencing. The Radar SDK runs on the user's phone, detecting when they start moving, classifying the trip type (e.g., walking, cycling, train, or driving an automobile), and sending a compact trip summary payload to our GCP API Gateway.
* **TomTom Routing API / OpenStreetMap:** Used to calculate precise transit distances or alternative route modeling (e.g., calculating exactly how much carbon was avoided by taking a 5-mile train route instead of a 5-mile driving route).

### C. Environmental Intelligence & Carbon Math

* **Electricity Maps API:** Provides real-time, highly granular grid carbon intensity data across Europe, North America, and parts of Asia. It updates dynamically based on the exact mix of solar, wind, coal, or gas feeding the regional power grid at that specific hour.
* **EPA eGRID API / OpenEmmissions API:** For regions without real-time tracking, these open APIs provide reliable, government-vetted regional emission factor lookups grouped by postal code or electrical subregion grid boundaries.

### D. Verification & Corporate Rewards

* **B-Corp API / Directory Services:** Used by our administration portal to dynamically verify that any brand applying to sponsor an impact pool possesses a valid, active B-Corporation certification.
* **Eden Reforest Projects / One Tree Planted Webhooks:** When a user redeems their earned "Leaves" to plant a physical tree, our backend triggers a secure API request to the non-profit partner to execute the planting allocation and return a tracking receipt link.
